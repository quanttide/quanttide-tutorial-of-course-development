# 如何制作一份课程蓝图

课程蓝图不是教案，不是大纲，也不是知识点的清单。它是隐藏在教学背后的那个核心思路——回答的不是"教什么"，而是"为什么这样教"。

这篇文章教你从零开始做出一份课程蓝图。

## 第一步：找到那个让新手困惑的操作

每一份课程蓝图都从一个问题开始：**初学者在使用这个工具/概念时，最具体的操作困惑是什么？**

这里的关键词是"具体操作"。不是"我不理解 Docker"，而是"为什么有镜像还要 docker run？"。不是"Git 好难"，而是"为什么本地 commit 了还要 push？"。这种"多此一举"的感觉，就是蓝图的入口。

怎么找到它？回想你自己初学时的困惑，或者观察新人第一次使用时卡住的地方。那个让你觉得"为什么要多这一步"的瞬间，就是它。

## 第二步：回到设计源头

找到困惑之后，不要直接解释操作，先往回走——回到这个东西被设计出来的那个原始场景。

问三个问题：

1. **这个东西是怎么诞生的？**——Linus 写 Git 是因为 BitKeeper 收回了授权，他面对的是上千个全球分布的贡献者。Docker 的设计者面对的是"环境不一致"这个运维噩梦。
2. **当时的替代方案是什么？**——Git 出现之前是 SVN（集中式），Docker 出现之前是虚拟机。没有对比就没有理解。
3. **新方案的核心决策是什么？**——Git 做了"分布式"这个决策，Docker 做了"镜像与容器分离"这个决策。一个决策，解释了后面所有的设计。

这三个问题回答清楚，设计源头就透了。

## 第三步：回到那个困惑

现在你理解了设计源头，再回头看那个困惑，答案自然浮现。

- "为什么本地 commit 了还要 push？"——commit 是对自己负责（保存到本地仓库），push 是对团队负责（同步到远程仓库）。两件事，两个动作。
- "为什么有镜像还要 docker run？"——镜像定义"跑什么"（只读模板），run 创建"怎么跑"（运行时实例）。一个模板可以实例化无数次。

困惑没有变，但它在学习者眼中的性质变了——从"多此一举"变成了"原来如此"。

## 第四步：分课时

课程蓝图不需要覆盖所有知识点。它只需要组织好两个课时：

**课时 1**：解释核心困惑，讲清设计源头，建立对比框架。这一课解决最大的认知障碍。

**课时 2**：延伸相关概念。学生有了课时 1 的基础，现在可以深入协作流程、多容器编排等进阶话题。

两个课时之间有一条清晰的逻辑链条——课时 2 是课时 1 的自然延伸，不是跳转。

## 一个完整的例子

以 Git 为例，走一遍完整流程。

**困惑**："为什么本地 commit 了还要 push？"——这是分布式 vs 集中式这个核心差异在日常操作中的体现。

**设计源头**：SVN 是集中式的，所有操作依赖中央服务器。Git 是分布式的，人人本地有完整仓库。Linus 这么设计是因为 Linux 内核有上千个全球分布的贡献者。

**回到困惑**：commit 是保存到本地，push 是同步到远程。两件事，两个动作。

**课时 1**：分布式 vs 集中式对比 + 三区操作（工作区、暂存区、本地仓库）

**课时 2**：分支概念 + 两种分支模式（feature branch vs trunk-based）+ 自动化

## 常见错误

- **从概念出发，不从困惑出发。** "今天我们来学习分布式版本控制的概念"——学生听完就困。要从"你有没有遇到过这个麻烦"开始。
- **试图覆盖所有知识点。** 一份蓝图只解决一个核心困惑。其他的可以留给别的蓝图或实战。
- **把蓝图写成讲课稿。** 蓝图是框架，不是逐字稿。它告诉老师要讲什么，不替老师讲课。
